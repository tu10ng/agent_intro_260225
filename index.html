Agent = 有手有脚的 AI

传统 AI

🗣️ 只会说话的顾问

  • "你应该这样做..."
  • "建议你考虑..."
  • 给建议，但不动手

AI Agent

💪 能动手干活的助理

  • "我来帮你做"
  • "已经完成了"
  • 理解需求，直接执行

Agent 的工作方式：循环

感知
→
思考
→
行动
→
观察结果
↩️

例子：点外卖

  1. 感知：看菜单，了解有什么
  2. 思考：选什么菜？价格合适吗？
  3. 行动：下单付款
  4. 观察：订单状态、配送进度
  5. 再思考：要不要催单？地址对吗？
停下来想一想：你日常工作中有哪些任务也是这种循环模式？

Agent 的组成

Agent = LLM + 工具 + 循环
Agent 就像一个"全能实习生"：
  • 🧠 LLM = 大脑：聪明，能理解你说的话
  • 🔧 工具 = 手脚：能操作电脑、读写文件、上网搜索
  • 🔄 循环 = 工作习惯：干一步，看一步，不对就调整

上下文 = Agent的"工作记忆"

比喻：上下文窗口 = 会议室白板的大小
  • 白板有限 → 写满了就得擦掉旧内容
  • 塞太多信息 → AI记不住前面说的，开始"发蒙"
核心警告：塞太多信息 → "专家变成新手bug制造机"

信息过载时，AI的输出质量会急剧下降

实操要点：

  • 每次只聚焦一个改进点
  • 复杂任务拆成小步骤
  • 新对话 = 清空白板重新开始

  • 本质：一个编程领域的 Agent
把前面说的 Agent 概念，具体化到编程场景

Claude Code 能干什么

用具体例子说：

  • 📖 读代码 - "这个函数是干嘛的？"
  • ✏️ 改代码 - "给这个函数加个参数校验"
  • 🏃 跑命令 - "运行测试看看有没有问题"
  • 🔍 搜文件 - "找到所有用了这个API的地方"
核心价值：你说人话，它干活

和 AI 沟通的技巧（一）

不是"写提示词"，是"和聪明的实习生沟通"

沟通三要素：

1. 背景

这是什么项目？

现在是什么状态？

2. 目标

你想达成什么？

什么算"做完了"？

3. 约束

有什么限制？不能改什么？有什么依赖？

和 AI 沟通的技巧（二）

渐进式沟通法：

❌ 一次性甩一大段需求

AI容易遗漏细节或理解偏差

✅ 分步骤描述

每步确认后再继续

上下文喂养技巧：
  • 告诉AI "先看一下xxx文件"
  • 主动提供相关代码片段
  • 说明这段代码和其他模块的关系

沟通实战示例

场景：让Claude Code重构一个函数

Plan模式 - 复杂任务必备

Plan模式 = 装修前先画图纸
  • 直接让装修队干活 → 改来改去，浪费材料
  • 先画图纸确认 → 一次到位，省时省钱

什么时候用：

  • 任务涉及多个文件
  • 不确定最佳实现方案
  • 项目后期的修改（避免牵一发动全身）
$ claude
> /plan 重构用户认证模块
# Claude会先分析，给出计划，你确认后再执行

Plan模式实战流程

1. 描述任务
→
2. AI分析现状
→
3. AI提出方案
4. 你确认/调整
→
5. AI执行
→
6. 验证结果

实操提示：

  • 可以要求AI列出会修改哪些文件
  • 可以要求AI解释为什么这样改
  • 不满意可以让AI重新规划
好处：避免AI为修局部bug改坏整体设计

调试技巧 - 日志是最佳"沟通语言"

为什么：AI分析结构化日志 >> 听你口述"哪里不对"
差的做法 
print("error")
好的做法 
logger.error(
  f"用户登录失败: "
  f"user_id={user_id}, "
  f"reason={e}, "
  f"timestamp={now}"
)
# 给AI调试时：
这是报错日志：[粘贴完整日志]
发生在执行xxx操作时。
请分析原因并修复。
快速检验：A还是B更适合给AI调试？
A: "登录有问题"    B: "日志显示user_id=123在14:30登录时返回401"

多Agent并行 - 大项目提效

多Agent = 多个实习生分工
  • 一个人又做用户又做订单又做支付 → 手忙脚乱
  • 三个人各管一摊，最后汇总 → 效率翻倍

适用场景：项目有多个独立模块

终端1: claude → 负责用户模块
终端2: claude → 负责订单模块
终端3: claude → 负责支付模块
最后手动合并/解决冲突
注意事项：任务要能独立 | 避免同时改同一个文件 | 每个Agent专注自己的模块

工具组合策略（一）- 工具对比

工具 擅长 不擅长
Grok/Perplexity 查最新资料、社区实践 理解你的代码
Claude Code 理解代码、执行修改 实时联网搜索
ChatGPT 头脑风暴、方案讨论 操作本地文件
Cursor/Copilot IDE内实时补全 复杂多文件重构
选择原则：用对的工具做对的事，不要指望一个工具搞定所有

工具组合策略（二）- 实战Workflow

1. 需求分析 → ChatGPT头脑风暴
↓
2. 技术选型 → Grok查最新实践
↓
3. 架构设计 → Claude /plan模式
↓
4. 分模块开发 → 多Agent并行
↓
5. 调试修复 → 日志 + Claude分析
↓
6. 代码审查 → Claude review
组合workflow：用Grok查最新实践 → 把信息喂给Claude Code → 执行具体实现

第三部分小结

复杂任务应该用什么模式？调试时最好给AI看什么？
要点回顾：
  • 复杂任务 → /plan 模式（先画图纸再装修）
  • 调试 → 给AI看日志（结构化信息 >> 口头描述）
  • 大项目 → 多Agent并行（多个实习生分工协作）
  • 工具组合：不同工具各有擅长，组合使用效果最佳

什么时候用 / 不用

✅ 适合

  • 重复性任务
  • 探索陌生代码库
  • 快速原型开发
  • 标准化的改动
  • 代码解释和学习

❌ 不太适合

  • 需要深度业务理解的决策
  • 高风险的核心逻辑修改
  • 需要团队讨论的架构设计
  • 涉及安全敏感的代码
你来判断：
"给100个文件批量改名" → 适合吗？
"设计核心支付架构" → 适合吗？

常见误区（上）

误区1：一次塞太多

症状：AI输出质量下降，开始胡说八道，前后矛盾

原因：上下文窗口（白板）写满了，旧信息被挤掉

✅ 解法：拆分任务，复杂任务新开对话，每次聚焦一个点
误区2：盲目信任

症状：AI改的代码没测就上线，结果出bug

原因：AI也会犯错，尤其是边界情况和业务逻辑

✅ 解法：永远跑一遍测试，关键代码人工review

常见误区（下）

误区3：不给上下文

症状：AI改错文件，或者理解错需求

原因：AI不知道你的项目结构和当前状态

✅ 解法：先让AI读相关文件，说明背景和约束
误区4：放弃太快

症状：AI第一次没做好就换方案或放弃

原因：AI需要反馈才能调整，不是一次就完美

✅ 解法：追问、给反馈、指出问题让它迭代调整

出问题怎么办 - 急救清单

AI越改越乱
git checkout . 回到上次commit → 新开对话，重新描述
AI不理解需求
给具体例子："输入xxx应该输出xxx" → 指定参考文件："按照xxx文件的风格"
AI改错地方
明确指定文件路径 → "只改xxx函数，不要动其他地方"
万能急救：git commit频繁提交，随时可以回退
