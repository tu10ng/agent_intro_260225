Agent 的组成

Agent = LLM + 工具 + 循环
Agent 就像一个"能干的研究生"：

（能力很强，但还没到博士/专家水平）

  • 🧠 LLM = 大脑：聪明，能理解你说的话
  • 🔧 工具 = 手脚：能操作电脑、读写文件、上网搜索
  • 🔄 循环 = 工作习惯：干一步，看一步，不对就调整

例子：点外卖

  1. 感知：看菜单，了解有什么
  2. 思考：选什么菜？价格合适吗？
  3. 行动：下单付款
  4. 观察：订单状态、配送进度
  5. 再思考：要不要催单？地址对吗？

Agent 的"疲劳"现象

想一想：
刚才说 Agent 像研究生，能循环干活。
但研究生连续干8小时会怎样？
  • 脑子越来越乱，开始犯低级错误
  • 前面说的事情忘了，需要反复提醒
  • 明明很聪明的人，表现得像新手
Agent 也有类似的问题：对话太长、塞太多信息 → AI也会"发蒙"

原因：上下文窗口的限制

比喻：上下文窗口 = 会议室白板的大小
  • 白板有限 → 写满了就得擦掉旧内容
  • 塞太多信息 → AI记不住前面说的，开始"发蒙"
核心警告：塞太多信息 → "专家变成新手bug制造机"

信息过载时，AI的输出质量会急剧下降

实操要点：

  • 每次只聚焦一个改进点
  • 复杂任务拆成小步骤
  • 新对话 = 清空白板重新开始

和 AI 沟通的技巧（一）

还记得Agent循环吗？ 感知 → 思考 → 行动 → 观察
问题来了：它"感知"的是什么？→ 是你给它的信息！
信息质量 → 决定思考质量 → 决定行动质量
想象你给研究生布置任务：
  • 不说背景 → 他不知道项目在做什么
  • 不说目标 → 他不知道做到什么程度算完
  • 不说约束 → 他可能改了不该改的地方

沟通三要素

1. 背景

这是什么项目？

现在是什么状态？

2. 目标

你想达成什么？

什么算"做完了"？

3. 约束

有什么限制？不能改什么？有什么依赖？

背景 ←→ 目标 ←→ 约束
  ↘       ↓       ↙
      AI的理解

想一想

回想一下：
上次AI给你的结果不满意，
是不是这三个里少了哪个？

常见问题：

  • 没说背景 → AI假设错误的上下文
  • 没说目标 → AI不知道做到什么程度
  • 没说约束 → AI改了不该改的东西

和 AI 沟通的技巧（二）- 渐进式

还记得Agent的工作方式吗？ 它是循环的——干一步、看一步。
我们的沟通也要配合这个循环：
给一步指令 → 看AI的理解 → 确认后再给下一步
❌ 一次性甩一大段需求

AI容易遗漏细节或理解偏差

✅ 分步骤描述

每步确认后再继续

渐进式沟通示例

例：重构一个复杂函数

第1轮："先看一下 user_service.py 这个文件"
↓ AI读完，确认它理解了结构
第2轮："里面的 process_user 函数太长了，你觉得可以拆成几个？"
↓ AI给出方案，你确认或调整
第3轮："按第二种方案来，先改 validate 部分"
↓ 逐步执行

新手 vs 老手的沟通方式

新手做法 老手做法
"帮我重构这个" "先读文件，告诉我你的理解"
一次说完所有需求 每步确认后再继续
不满意就放弃 追问"哪里不对，再试一次"
等AI全做完再看 中途就检查方向对不对
核心：把大任务拆成小步骤，每步都确认

开发优先级 - 特性完整法则

前面说了沟通技巧，现在问题来了：
任务这么多，怎么拆？
答案：以特性为单位，每个做到最好。
特性完整法则 = 一个组件100%完成，再开始下一个
（而不是多个组件各做80%再统一收尾）
❌ 反面案例：
登录、注册、找回密码都做了80%
→ AI 后续改登录时出 bug，无法定位是登录的问题还是影响了注册

为什么要做完整？

想象：每次对话都是新实习生接手
他只能看代码本身，不知道你之前的口头交代
→ 所以代码必须"自解释"，不能留半成品
实操：按独立特性拆分，每个含功能+测试+错误处理

Plan模式 - 项目后期必备

AI 也有"偷懒"倾向

项目后期让 AI 修一个小 bug，它可能会：

为了快速"修好" → 找最省力方案 → 把你原有设计改得面目全非

装修改电路 - 敷衍版
插座松了？
→ 拆掉，拉根明线
→ "修好了！"
（结果：墙上多了条丑线）
装修改电路 - 负责版
先看懂原来的电路图
→ 找到真正的问题
→ 不破坏整体布线
（Plan模式 = 让AI先说思路）

识别AI"走捷径"的信号

新手看比喻 老手看代码
AI说"我把这个函数简化了" 把你的策略模式改成一堆 if-else
AI说"我移除了不必要的抽象" 删掉你精心设计的接口层
AI说"我用更直接的方式" 硬编码替代你的配置系统

什么时候必须用Plan：

  • 项目后期的任何修改（核心！代码量越大越要review）
  • 涉及已有架构的改动
  • AI的修改范围超出你的预期

调试技巧 - 为什么要给AI看日志？

日志是AI的"低成本高回报"输入
1万行日志 ≈ 几千token（重复格式被压缩）
→ 你觉得"太长了"，AI觉得"刚刚好"
人工分析 AI分析
逐行看，容易漏 全量扫描，不遗漏
日志和代码来回切换 同时看，自动关联
两份日志对比很累 "有什么不同？"秒答
时间线长了头疼 自动重建事件链

调试技巧 - 怎么让AI帮你看日志

场景：线上有个偶发bug，日志量很大

# 把日志喂给AI：
这是最近1小时的日志，有3次报错。
请找出：1) 报错前发生了什么 2) 三次是否有共同点
AI能做而你懒得做的事：
- 在1000行日志里找那3次报错的上下文
- 对比三次报错，发现"都是在调用XX接口后5秒内"

多Agent并行 - 大项目提效

前面一直是一个Claude在干活。
但如果项目很大呢？
一个研究生 vs 三个研究生分工
  • 一个人又做用户又做订单又做支付 → 手忙脚乱
  • 三个人各管一摊，最后汇总 → 效率翻倍

（但需要协调，不能让他们打架）

终端1: claude → 负责用户模块
终端2: claude → 负责订单模块
终端3: claude → 负责支付模块
最后手动合并/解决冲突

判断：哪个适合多Agent？

你来判断：以下哪个适合多Agent？
A. 重构一个300行的函数
B. 三个独立模块同时开发
C. 修复一个涉及5个文件的bug
答案：B最适合
A太小没必要 → C文件有依赖会冲突 → B模块独立，适合分工

多Agent注意事项

⚠️ 关键注意事项：
  1. 任务要能独立 → 否则Agent之间会"打架"
  2. 避免同时改同一个文件 → Git冲突
  3. 每个Agent专注自己的模块 → 别让A去改B的代码
  4. 最后手动review合并点 → 人工把关接口
简单判断：如果两个任务会改同一个文件 → 不要并行

什么时候用 / 不用

判断原则：这个任务适合"循环式、渐进式"地完成吗？

✅ 适合

  • 重复性任务（批量改动）
  • 探索陌生代码库
  • 快速原型开发
  • 标准化、可测试的改动
  • 代码解释和学习

❌ 不太适合

  • 需要深度业务理解的决策
  • 高风险且难以验证的改动
  • 需要团队讨论的架构设计
  • 涉及安全敏感的代码
灰色地带判断（想一想再看答案）：
"修改核心支付逻辑的一个参数校验" → ？
（✅ 适合 — 改动明确、可测试、用Plan模式review）
"在已有架构上加一个新的支付渠道" → ？
（⚠️ 谨慎 — 改动范围大，建议先Plan+人工review架构影响）

常见误区（上）

在看答案之前，想一想：你觉得新手最容易犯哪种错？
误区1：一次塞太多

症状：AI输出质量下降，开始胡说八道，前后矛盾

原因：上下文窗口（白板）写满了，旧信息被挤掉

↑ 回顾：还记得"会议室白板"吗？这就是白板被写满的后果
✅ 解法：拆分任务，复杂任务新开对话，每次聚焦一个点
误区2：盲目信任

症状：AI改的代码没测就上线，结果出bug

原因：AI也会犯错，尤其是边界情况和业务逻辑

✅ 解法：永远跑一遍测试，关键代码人工review

常见误区（下）

误区3：不给上下文

症状：AI改错文件，或者理解错需求

原因：AI不知道你的项目结构和当前状态

↑ 回顾：还记得"沟通三要素"吗？这就是缺了"背景"
✅ 解法：先让AI读相关文件，说明背景和约束
误区4：放弃太快

症状：AI第一次没做好就换方案或放弃

原因：AI需要反馈才能调整，不是一次就完美

↑ 回顾：Agent是循环的——干一步看一步，需要迭代
✅ 解法：追问、给反馈、指出问题让它迭代调整
